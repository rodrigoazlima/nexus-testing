# QA Feedback: `agent.json` now exists right after install — test's own documented assumption is stale

**Reported by:** `tests/pipeline/stage-setup-agent-config.spec.ts:39` (`agent.json does not exist right after install (runner hasn't run yet)`)
**Symptom:** `fs.readFile(agentJsonPath).catch(() => null)` returned a full, valid `agent.json` for `vision-agent` immediately after a fresh install — the test expects `null`.

This document only analyzes — no code was changed while producing it. This is a test-harness/assumption bug, not a Nexus defect: the installer's actual behavior is intentional and self-documented, it just no longer matches this spec's comment.

---

## Root Cause

The spec's own comment (`stage-setup-agent-config.spec.ts:12-17`) states the assumption plainly:

> `agent.json` itself isn't synthesized at install time — runner.py only writes it on the daemon's first initialization run — so the existence check has to wait out that first run...

This is no longer true. `system/ops/setup-service.ps1:1039-1048` explicitly runs config synthesis **during install**, before the daemon/scheduler ever starts:

```powershell
# agent.json is gitignored (matches the repo's blanket *.json rule), so a fresh
# ... never run. nexus.runner --ensure-config synthesizes any missing agent.json from
Log "Ensuring agents/*/agent.json exist (synthesizing missing ones from registry.yaml)..."
```

That invokes `python -m nexus.runner --ensure-config`, handled in `system/src/nexus/runtime/cli.py:41-43`:

```python
if args.ensure_config:
    ensure_agent_jsons(log)
    sys.exit(0)
```

which calls `ensure_agent_jsons()` (`system/src/nexus/runtime/registry_yaml.py:80-99`) — itself explicit about why:

```python
"""Self-heal: create agent.json for any active agent missing one.

agent.json is gitignored (matches the blanket `*.json` rule) so a fresh
clone/install starts with zero of them, and the scheduler discovers zero
tasks. Regenerate the missing ones from registry.yaml defaults so the
pipeline works out of the box; a hand-edited agent.json is never touched.
"""
for agent_name, entry in (load_registry_dict().get("agents") or {}).items():
    if entry.get("status") != "active":
        continue
    agent_json = AGENTS_DIR / agent_name / "agent.json"
    if agent_json.exists():
        continue
    payload = synthesize_agent_json(agent_name, entry)
    ...
    agent_json.write_text(json.dumps(payload, indent=2), encoding="utf-8")
```

`synthesize_agent_json()` (`registry_yaml.py:43-77`) builds the payload straight from `registry.yaml` — this exactly reproduces the file actually found on disk for vision right after install:

```json
{
  "tasks": {
    "vision-agent": {
      "intervalSeconds": 300,
      "description": "vision agent",
      "dispatch": { "type": "cli", "cli": { "command": "python", "args": ["agents/vision/tools/classify_images.py"], "cwd": "project_root", "timeout_seconds": 1800 } }
    }
  }
}
```

So the sequence is: `npm test` → `global-setup.ts` → `installFresh()` runs `setup-service.ps1` → the script's own config-synthesis step calls `--ensure-config` → `agent.json` is written for every active agent **before Playwright ever gets control back**. There is no window in this test repo's lifecycle where `agent.json` is absent — the "right after install" state the second test tries to catch doesn't exist anymore.

(Aside, not the cause of this failure but worth noting for whoever touches this next: `CLAUDE.md` in the Nexus repo itself says `agent.json` is *"regenerated by the maintenance worker"*, and a second, independent scaffold — `_AGENT_JSON_SPECS` in `system/src/nexus/workers/maintenance.py` — produces a *different* shape (`claude-api` dispatch, not `cli`) for the same purpose. Both mechanisms only act "if missing," so whichever runs first wins; today that's install-time `--ensure-config`, which is why the on-disk file is the `cli` shape, not the `claude-api` one.)

---

## Steps to Reproduce

1. Run `npm test` (or `npm run test:only -- tests/pipeline/stage-setup-agent-config.spec.ts`) from a clean state.
2. Let `global-setup.ts` finish installing (`clearInstall()` → `installFresh()` → `setup-service.ps1`).
3. Immediately check `.testing/nexus/agents/vision/agent.json` (or any other active agent's) — it already exists, populated from `registry.yaml` defaults, before the runner's scheduler has run a single cycle.

---

## Proposed Solution

Update `stage-setup-agent-config.spec.ts` to match current, intentional installer behavior rather than the stale assumption:

- Drop (or invert) the second test (`agent.json does not exist right after install`) — install-time synthesis via `--ensure-config` is deliberate, documented, self-healing behavior in `setup-service.ps1`/`registry_yaml.py`, not an accident to guard against.
- The third test (`every agent.json exists and agrees with the registry after the runner's first initialization run`) is likely still valid in spirit but its premise ("first initialization run" writes it) should be corrected to "install-time `--ensure-config` writes it, and it must already be present and correct without waiting on any runner cycle at all" — which would let that assertion turn into a fast, non-polling check instead of the current `toPass({ timeout: 2 * 60_000, ... })` wait.
- If there's a genuine desire to test "agent.json doesn't exist before the config-synthesis step specifically," that would need to be checked at a point *inside* `setup-service.ps1`'s own run (not observable from outside via Playwright today), or by intentionally invoking `nexus.runner --ensure-config` a second time against a directory where it was deleted first — a different, more explicit test than what's here now.
